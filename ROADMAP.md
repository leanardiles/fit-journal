# FitJournal Roadmap

FitJournal is a personal fitness tracking application. This roadmap tracks planned features, technical debt, and the path from local development to AWS production deployment.

**Last updated:** 8 July 2026

---

## Priority & Effort Legend

| Priority | Meaning |

| 🟠 P1 | High — significant feature or blocker to deployment |
| 🟡 P2 | Medium — quality improvement, not urgent |

- **[Backend / Web] Routine types — per_muscle, manual, mixed (full refactor)** — Replaced the single muscle-groups-per-day routine model with a first-class training-day model supporting three day types. New tables (`training_days`, `training_day_muscles`, `training_day_exercises`, `exercises_in_routine`) added additively — `create_all` builds them on cold start, no migration — while the old `routine_days`/`routine_muscles_per_day` tables and `exercise_is_in_routine` flag are left dead (cleanup backlogged). **Day types:** *per_muscle* = a selected pool of exercises per muscle with a per-session count, where generation rotates by picking the least-performed from the pool (rotation counter `times_performed` lives on `exercises_in_routine`, one shared row per exercise); *manual* = an exact exercise list taken whole; *mixed* = type chosen per day. **Reconcile-on-save:** editing a routine preserves the rotation counter for exercises that stay, deletes rows for removed ones, and enters newly-added exercises at the *minimum* counter among surviving same-muscle exercises (so a new exercise joins rotation as "next up", not force-picked or stale) — the specific behaviour requested so adding one exercise never resets the others. **Backend:** rewrote `create_or_update_routine` (whole-payload validation before any write — contiguous day numbers, per-muscle pool coverage, manual needs an exercise, ownership — then atomic clean-slate replace + reconcile), `get_routine`, `generate_next_workout` + helpers, and `complete_workout` (days-per-week now derived from training-day count; completion increments the pool counter). **Web:** rebuilt the routine page (day-count → routine-type → per-day editors; per_muscle muscle-buttons + expandable pool with counts, manual Exercises-page-style tabs + checkbox list, mixed per-day type toggle with confirm-before-clear on switch; edit mode prefills and locks routine-level type while keeping per-day toggles live for mixed; current-routine view shows one column per day, per-day type tags, and per_muscle days collapsed to muscle+count with a "see pool" expander). ## 🧭 Guiding Principles

A few standing rules for how this project evolves.

- **Schema changes need backups first.** Snapshot RDS before any `ALTER TABLE` or major version change.
- **Local dev uses local MySQL.** Start the MySQL80 service before running uvicorn; `.env` points at `127.0.0.1`. RDS is production-only now.
- **Repo is the source of truth.** Deployed code always derives from a commit — never edit code directly in the Lambda console.
- **Deploy is a separate, deliberate step.** `git push` saves to GitHub; deploying runs the GitHub Actions workflow (currently manual). Don't ship a link/feature before the page/endpoint it needs exists.
- **Compiled deps must target Lambda's runtime.** Lambda is Amazon Linux 2 (glibc 2.26); CI installs `manylinux2014` wheels so binaries like bcrypt load.
- **Page links use the `/web/` prefix.** `/web/routine`, `/web/dashboard`, etc. — bare `/routine` 404s. API endpoints (`/routine/{id}`) are separate; don't confuse the two. (Case-sensitivity bites here too — Linux/prod is case-sensitive, Windows-local isn't.)
- **Branch per feature.** Don't commit unrelated changes together.
- **Theme variables, not hardcoded colors.** Any `color: #ccc` is a future bug.
- **Migrations over recreations.** Preserve user data wherever possible.
- **Document the why, not the what.** Code shows what; commits and ROADMAP show why.
- **Every roadmap item declares its Scope.** New items go in the planning tables with a Scope value (`Mobile`, `Web`, `Backend`, `Mobile / Web`, or `All`). New Done entries use an inline `[Mobile]` / `[Web]` / `[Backend]` prefix tag. Existing pre-June-2026 Done entries left untagged.
